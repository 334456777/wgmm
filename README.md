# 🎯 智能视频监控系统 - 基于WGMM机器学习算法

## 🚀 项目概述

这是一个基于**加权高斯混合模型 (WGMM)** 的智能B站视频监控系统，采用先进的机器学习算法自动学习UP主的发布规律，实现自适应的智能监控频率调整。系统能够显著提高监控效率，减少不必要的网络请求，同时确保及时捕获新发布的视频。


**2. 编辑 `.env` 文件，填入您的实际配置：**
```bash

# Bark 推送通知设备密钥
```

**3. 如何获取这些配置：**

- **GITHUB_TOKEN**: 
  1. 访问 https://github.com/settings/tokens
  2. 点击 "Generate new token (classic)"
  3. 勾选 `gist` 权限
  4. 生成并复制 token

- **BARK_DEVICE_KEY**: 
  1. 在 iOS 设备上安装 Bark App
  2. 打开 App，复制设备密钥（通常是一串随机字符）

- **GIST_ID**: 
  1. 访问 https://gist.github.com
  2. 创建一个新的 Gist（可以是空的）
  3. 从 URL 中复制 Gist ID（如：`e5e6ead253b538ef71e31711744541f2`）

- **BILIBILI_UID**:
  1. 打开要监控的UP主的B站主页
  2. 从 URL 中获取 UID（如：`https://space.bilibili.com/123456789` 中的 `123456789`）
  3. 或在UP主主页点击"个人中心"查看 UID

- **记忆衰退模拟**：近期事件权重更高，远期事件逐渐被"遗忘"，适应习惯变化
- **相似度计算**：高斯核函数精确测量时间点相似性，实现精准预测

### 🎯 **低活跃期智能调整**
- **双维度活跃度分析**：综合考虑小时维度和星期维度的历史发布频率
- **线性映射优化**：活跃度0时检查间隔延长4倍，活跃度满值时保持正常频率

## 🎯 WGMM智能算法深度解析

### 💡 算法设计理念

基于**时间序列周期性模式识别**的机器学习方法，将视频发布行为建模为在时间维度上的概率分布问题。通过历史数据学习UP主的发布习惯，预测未来时间点的发布概率，实现智能化的监控频率调整。

### 🔬 核心数学模型

#### 1. 周期性时间特征编码
将线性时间转换为高维周期性特征空间，解决时间边界跳跃问题：

```mathematical
日周期特征：
    f_day(t) = (sin(2π × t_day/86400), cos(2π × t_day/86400))

周周期特征：  
    f_week(t) = (sin(2π × t_week/604800), cos(2π × t_week/604800))

其中：
    t_day = 当日秒数 (0-86399)
    t_week = 当周秒数 (0-604799)
```

#### 2. 时间相似性度量
使用欧几里得距离在高维周期性空间中测量时间点相似性：

```mathematical
周期性距离² = D_week² + 0.5 × D_day²

其中：
    D_week² = ||f_week(t_current) - f_week(t_history)||²
    D_day² = ||f_day(t_current) - f_day(t_history)||²
```

#### 3. 高斯核相似性函数
将距离转换为相似性概率，控制时间模式匹配的严格程度：

```mathematical
Similarity(t_current, t_history) = exp(-Distance²/(2σ²))

参数配置：
    σ = 0.8  # 时间容忍度控制参数
    σ越小 → 匹配越严格，仅响应非常相似的时间模式
    σ越大 → 匹配越宽松，对时间差异更包容
```

#### 4. 指数时间衰减权重
模拟人类记忆衰退特性，近期事件影响更大：

```mathematical
Weight(t_event, t_current) = exp(-λ × Δt_hours)

其中：
    λ = 0.0001  # 遗忘速度控制参数
    Δt_hours = (t_current - t_event) / 3600  # 事件年龄（小时）
```

#### 5. WGMM热力得分聚合
综合所有历史事件对当前时间的影响：

```mathematical
Heat_Score = (Σ Weight_i × Similarity_i) / N_events

其中：
    N_events = 有效历史事件总数
    取值范围：[0, 1]，1表示最高发布概率
```

### 🎯 低活跃期智能调优

#### 双维度活跃度分析算法
```mathematical
小时活跃度 = 当前小时发布次数 / (总发布次数/24)
星期活跃度 = 当前星期发布次数 / (总发布次数/7)
综合活跃度 = 小时活跃度 + 星期活跃度

活跃度范围限制：Clamped_Activity = max(0, min(综合活跃度, 2.0))
```

#### 线性频率调整映射
```mathematical
活跃度调整倍数 = 4.0 - 1.5 × Clamped_Activity

映射效果：
    活跃度 = 0.0 → 倍数 = 4.0 (检查间隔延长4倍)
    活跃度 = 1.0 → 倍数 = 2.5 (检查间隔延长2.5倍)  
    活跃度 = 2.0 → 倍数 = 1.0 (正常检查频率)
```

### 🎯 关键文件自动生成详解

#### `local_known.txt` - 本地已知URL列表

**自动生成机制：**
1. **程序启动时**：`__init__()` 调用 `load_known_urls()`
   - 如果文件不存在 → 创建空文件并保存
   - 如果文件存在 → 读取内容到内存

2. **Gist同步时**：`sync_urls_from_gist()`
   - 从Gist获取URL列表 → 更新 `known_urls`
   - **立即保存**到 `local_known.txt`

3. **发现新视频时**：`run_monitor()`
   - 更新 `known_urls` → **立即保存**到文件

4. **程序退出时**：`signal_handler()`
   - 通过 `save_known_urls()` 保存当前状态

**文件格式：**
```
https://www.bilibili.com/video/BV123456789
https://www.bilibili.com/video/BV987654321
https://www.bilibili.com/video/BV111111111
```

**首次运行自动生成：**
```python
# 程序流程
```

### 📊 日志格式说明

#### 星号视频通知格式

当发现视频时，日志会使用星号 `*` 表示视频数量，格式如下：

```
*** **
```

**含义解读：**
- **第一组星号** (`***`): Gist 未备份的视频总数（3个）
- **空格分隔**
- **第二组星号** (`**`): 其中真正的新视频数量（2个）

**实际示例：**

```log
# 场景1: 发现3个新视频（全部需要备份）
2026-01-07 18:46:39 - INFO - 检查开始                  <--
2026-01-07 18:47:50 - INFO - 预检查完成 - 预测检查: 无新内容 快速检查: 发现新内容
*** ***
2026-01-07 18:47:50 - INFO - WGMM调频 - 轮询间隔: 17 分钟 30 秒

# 场景2: Gist未更新，3个未备份但都是旧视频（提醒备份）
2026-01-07 19:05:20 - INFO - 检查开始                  <--
*** 
2026-01-07 19:05:20 - INFO - WGMM调频 - 轮询间隔: 1 小时 25 分钟

# 场景3: 5个未备份，其中2个是新视频
2026-01-07 20:30:00 - INFO - 检查开始                  <--
***** **
2026-01-07 20:30:00 - INFO - WGMM调频 - 轮询间隔: 15 分钟
```

**快速理解：**
- `*** ***` = 3个未备份，3个新视频（都需要备份）
- `*** **` = 3个未备份，2个新视频（有1个旧视频还没备份）
- `*** ` = 3个未备份，0个新视频（都是旧视频，提醒用户去备份）
- `*****` = 5个未备份，0个新视频（第二组为空）

**设计原理：**
- **双层检测机制**：区分"Gist未同步"和"真正的新视频"
- **避免误判**：Gist更新延迟不会影响 WGMM 算法的惩罚机制
- **提醒备份**：即使不是新视频，也会通知用户有视频需要备份到 Gist

## 🧠 WGMM智能特性详解

### 自动学习系统

**历史数据管理：**
- **自动数据生成**：首次运行时自动获取所有历史视频的发布时间
- **智能数据修复**：读取失败时自动重新生成（最多3次尝试）
- **数据完整性保障**：保留所有时间戳，包括同一天发布的多个视频
- **去重保序**：自动去除重复时间戳但保持时间顺序

**WGMM算法核心：**
- **周期性学习**：自动识别UP主的发布规律（如每周固定时间）
- **时间衰减**：近期发布的视频对预测影响更大
- **相似性计算**：使用高斯核函数计算时间点相似度
- **动态基础间隔**：根据历史平均发布间隔调整基础预测
- **热力得分**：实时计算当前时间点的发布概率

### 智能预测效果

**算法优势：**
```
传统固定间隔    ❌ 每1小时检查一次（效率低）
简单统计算法    ❌ 基于小时/星期简单统计（不准确）
WGMM智能算法   ✅ 基于复杂时间模式学习（高效准确）
```

**实际效果示例：**
- **高活跃时段**：周三下午2点（历史发布较多）→ 5分钟检查一次
- **中等活跃时段**：工作日上午 → 30分钟检查一次  
- **低活跃时段**：深夜时段 → 6小时检查一次
- **长期空白期**：UP主休息期 → 数天到数周检查一次

### 快速预检查机制

避免重复的完整检查：

- **视频ID对比**：快速检查最新视频ID是否变化
- **分片检测**：检查现有视频是否增加新分片
- **并行处理**：多线程并行获取分片信息
- **智能缓存**：减少不必要的网络请求

### 智能频率调节算法

程序使用基于历史数据的概率权重算法：

**1. 自动历史数据管理**
- **首次运行检测**：检测 `mtime.txt` 是否存在，不存在时自动创建
- **元信息获取**：使用 `yt-dlp --write-info-json` 获取所有视频的发布时间
- **时间戳提取**：优先使用 `timestamp` 字段，备选 `upload_date` 字段
- **多重重试**：生成失败时最多重试3次，失败后发送 Bark 通知
- **完整性保障**：保留所有时间戳，包括同时发布的多个视频

**2. 数据处理优化**
- **智能统计**：使用 `mtime.txt` 的实际行数作为总发布次数
- **故障恢复**：读取失败时自动重新生成历史数据
- **容错处理**：多种备选方案确保算法稳定运行
- **数据完整性**：不去重时间戳，确保统计真实反映发布频率

**3. 时间分布统计**
- 统计不同星期几的更新频率
- 统计不同小时的更新频率

**4. 高精度概率权重计算**
```python
# 总发布次数计算（使用mtime.txt实际行数）
总发布次数 = len(mtime.txt中的所有行)

# 星期概率权重
星期权重 = 当前星期更新次数 × 7 × 1,000 / 总发布次数

# 小时概率权重  
小时权重 = 当前小时更新次数 × 24 × 1,000 / 总发布次数

# 综合权重
综合权重 = 星期权重 × 小时权重 / 1,000
```

**5. 智能频率策略**
- **权重 > 0**：最终频率 = 基础频率 × 1,000 / 权重
- **权重 = 0**：使用最大间隔（4321分钟≈3天）

**6. 边界限制**
- 最小检查频率：30分钟
- 最大检查频率：4321分钟（约3天）

**数据质量保障：**
- **自动生成**：首次运行时自动创建完整的历史数据
- **多重重试**：文件生成或读取失败时自动重试多次
- **实时通知**：重要错误自动发送 Bark 通知
- **完整统计**：保留所有时间戳记录，确保统计准确性
- **容错机制**：多重备份策略确保算法稳定运行

## ❓ 常见问题解答 (FAQ)

### Q1: 为什么算法会形成3天的检查间隔？是硬编码的吗？

**A: 不是硬编码，而是WGMM算法通过数学计算自然涌现的结果。**

#### 算法没有硬编码"3天"

关键配置：
```python
lookahead_days = 15  # 向前扫描15天
sigma_week = 1.0      # 星期维度的高斯核宽度
```

算法会在**未来15天内扫描**，寻找最佳检查时间点。没有硬编码"3天"这个值。

#### 星期维度的数学编码

```python
weekday = (days_since_epoch + 3) % 7
# 0=周一, 1=周二, ..., 5=周六, 6=周日
```

将星期几转换为**周期性特征**（sin/cos 编码）：

```python
week_sin = sin(2π × weekday / 7)
week_cos = cos(2π × weekday / 7)
```

这样，**周六和周日**在特征空间中相邻，**周一到周五**形成另一簇。

#### 高斯核相似性计算

对于任意两个时间点 t1 和 t2：

```python
week_distance² = (week_sin₁ - week_sin₂)² + (week_cos₁ - week_cos₂)²
similarity = exp(-week_distance² / (2 × sigma_week²))
```

由于 `sigma_week = 1.0`：

| 星期几关系 | 距离 | 相似度 |
|-----------|------|--------|
| 同一天 | 0.0 | 1.000 |
| 相邻天（如周日→周一） | ~0.9 | **0.606** |
| 相隔2天 | ~1.8 | 0.135 |
| 相隔3天 | ~2.6 | 0.011 |
| 相隔4天 | ~3.5 | 0.000 |

#### 为什么会产生3天间隔？

从配置数据分析：

```json
"dimension_weights": {
  "week": 0.672,        // ← 最高权重！
  "month_week": 0.590,
  "day": 0.402,
  "year_month": 0.336
}
```

**数学含义**：
- 算法学习到**星期模式最重要**
- 361条历史数据分布在所有星期几
- 但由于方差极大（719,643,325,024 秒²），发布时间高度不规律

**算法的推理过程**：
```
1. 当前得分很低 → 基础间隔延长
2. 没有找到明确的峰值（best_peak_score <= 0.6）
3. 使用 base_frequency_sec 作为间隔
4. 经过低活跃期调整 → 最终约3天
```

#### 3天的数学意义

**3天 = 覆盖一周中大部分可能的发布时间**

如果今天检查：
- **今天**：覆盖可能的发布
- **明天**：相邻星期（相似度 0.606）
- **后天**：跨度稍大（相似度 0.135）
- **第3天**：覆盖另一极端（如工作日→周末或反之，相似度 0.011）

**这不是巧合**，而是因为：
- **星期维度权重最高**（0.672）
- **sigma_week = 1.0** 意味着相邻天的相似度 = 0.606
- **3天间隔** 确保以较高相似度覆盖工作日和周末

#### 数学验证：计算最佳间隔

假设今天是周四（weekday = 3），检查未来7天的相似度：

```
间隔 | 星期 |   与周四相似度 | 累计覆盖
--------------------------------------------------
 0天 | 周四 |       1.0000 |   1.0000
 1天 | 周五 |       0.6065 |   1.6065
 2天 | 周六 |       0.1353 |   1.7419
 3天 | 周日 |       0.0111 |   1.7530
 4天 | 周一 |       0.0111 |   1.7641
 5天 | 周二 |       0.1353 |   1.8994
 6天 | 周三 |       0.6065 |   2.5059
```

**结论**：
- 3天后累计相似度 = 1.75
- 4天后累计相似度 = 1.76（收益递减）
- **3天是性价比最高的间隔**

#### 美妙之处

3天间隔**自然涌现**自：
- 你的数据特点（361条分布在所有星期几）
- 学习到的维度权重（week=0.672最高）
- 高斯核的数学性质（sigma=1.0）

这就是 WGMM 算法的智能之处——**从数据中学习模式，而非预设规则**。

### Q2: WGMM算法的核心参数有哪些？如何调优？

**A: 算法有三大类核心参数，均可针对不同UP主类型进行优化。**

#### 核心参数配置

```python
# 时间维度的 sigma（高斯核宽度）
sigma_day = 1.0
sigma_week = 1.0
sigma_month_week = 1.5
sigma_year_month = 2.0

# Lambda 参数（遗忘速度）
lambda_min = 0.00005
lambda_base = 0.0001
lambda_max = 0.0005

# 其他参数
mapping_curve = 2.0          # 评分映射曲线
resistance_coefficient = 0.8 # 阻力系数
lookahead_days = 15          # 前瞻扫描天数
learning_rate = 0.1          # 维度权重学习率
min_history_count = 10       # 最小历史数据量
```

#### 针对不同UP主类型的优化策略

**高频规律型UP主（每日发布）：**
```python
SIGMA = 0.6              # 更严格的时间匹配
LAMBDA = 0.0002          # 稍快的记忆遗忘，适应近期变化
DEFAULT_INTERVAL = 1800  # 降低基础间隔至30分钟
```

**中频模式型UP主（每周2-3次）：**
```python
SIGMA = 0.8              # 平衡的时间容忍度（默认）
LAMBDA = 0.0001          # 标准记忆衰减速度
DEFAULT_INTERVAL = 3600  # 标准1小时基础间隔
```

**低频随机型UP主（每月1-2次）：**
```python
SIGMA = 1.2              # 更宽松的时间匹配
LAMBDA = 0.00005         # 更慢的记忆遗忘，保留长期模式

```
