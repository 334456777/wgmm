# ADR 001: 保持 Python 实现，不迁移到 Go

## 状态
已采纳 (Accepted)

## 日期
2026-01-21

## 背景 (Context)

### 当前技术栈
- **语言**: Python 3.14+
- **核心依赖**: NumPy (科学计算), requests (HTTP), yt-dlp (视频信息提取)
- **代码规模**: monitor.py 主程序 2,223 行，54 个方法

### 性能分析
当前 Python+NumPy 方案性能已经足够优秀：

| 组件 | 性能 | 说明 |
|------|------|------|
| **WGMM 算法核心** | ~10ms | NumPy 向量化计算 |
| **时间特征编码** | ~1ms | 1000条历史数据 |
| **批量评分计算** | ~5ms | 300个时间点预测 |
| **三层检测** | ~2s | 主要耗时在 yt-dlp I/O |
| **内存占用** | <10MB | 非常高效 |

**关键发现**: 算法计算(10ms) << I/O等待(2s)，**性能瓶颈在I/O而非计算**

### Go 迁移方案对比

| 方案 | 算法延迟 | vs Python | 开发周期 | 部署复杂度 |
|------|---------|-----------|---------|-----------|
| **纯 Go 标准库** | 150-200ms | 慢15-20倍 | 15-18周 | 低（单一二进制） |
| **Go + gonum** | 40-60ms | 慢4-6倍 | 11-14周 | 中（需CGO） |
| **混合架构** | 11-12ms | 慢1.1倍 | 8-11周 | 高（双进程） |
| **Python+NumPy** | **~10ms** | **基准** | **已完成** | 中（需依赖管理） |

**结论**: 任何 Go 方案都无法在性能和复杂度上同时超越 Python

### 系统耗时分解
```
总检查周期: ~3600秒 (1小时)
├── WGMM 算法计算: 0.010秒 (0.0003%)
├── yt-dlp I/O 等待: 2秒 (0.06%)
├── 网络请求: 0.5秒 (0.014%)
└── 睡眠等待: 3597.49秒 (99.92%)
```

**关键发现**:
- 算法计算占比 **0.0003%**
- 即使优化到 0ms，总性能提升 **0.0003%**
- **性能瓶颈在等待，而非计算**

## 决策 (Decision)

**不迁移到 Go 语言，保持当前 Python + NumPy 实现**

## 理由 (Rationale)

### 1. 性能层面
- ✅ 当前方案在算法性能上已达最优（~10ms）
- ✅ Go 方案无法提供实质性的性能提升（最慢20倍，最快慢10%）
- ✅ 性能瓶颈在 I/O 而非计算，优化算法无意义

### 2. 代码质量层面
当前 `monitor.py` (2,223行，54个方法) 的优势：

- ✅ **中文注释完善**: 所有函数都有详细的 docstring 和行内注释
- ✅ **算法实现成熟**: WGMM 算法经过充分验证
- ✅ **错误处理完善**: 多层异常捕获、重试机制、降级策略
- ✅ **三层检测架构**: 节省 90%+ 网络请求
- ✅ **并发优化**: ThreadPoolExecutor 并行处理
- ✅ **系统集成**: systemd 服务、环境变量管理

迁移风险：
- ❌ 丢失成熟的算法实现
- ❌ 引入新的 bug
- ❌ 需要重写 4000-5500 行代码
- ❌ 测试和调试成本高

### 3. ROI 分析

| 方案 | 性能提升 | 开发成本 | 维护成本 | 风险 | ROI |
|------|---------|---------|---------|------|-----|
| Go 标准库 | -150ms | 15-18周 | 高 | 高 | ❌ 负 |
| Go + gonum | -40ms | 11-14周 | 中 | 中 | ❌ 负 |
| 混合架构 | -1ms | 8-11周 | 高 | 中 | ❌ 负 |
| **保持 Python** | **基准** | **0周** | **低** | **无** | **✅ 最优** |

## 后果 (Consequences)

### 正面影响
- ✅ 保持代码的简洁性和可维护性
- ✅ 无需重写和测试，节省大量开发时间
- ✅ NumPy 在科学计算领域无可替代
- ✅ Python 代码质量高，无技术债务

### 负面影响
- ⚠️ 依赖 Python 环境和虚拟环境管理
- ⚠️ 部署需要安装依赖（但已通过 .venv 解决）

### 中性影响
- 📝 单体架构（2,223行）对于本项目的复杂度是合适的

## 替代方案（非必需）

如果确实希望改善部署体验，可以考虑：

1. **性能监控**: 添加 Prometheus metrics
2. **Docker 容器化**: 环境隔离和依赖打包
3. **PyInstaller 打包**: 单一可执行文件（不推荐，会破坏灵活性）
